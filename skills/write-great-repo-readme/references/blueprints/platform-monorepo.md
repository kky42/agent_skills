# Platform, Monorepo, or Multi-Component System

Recommended order:

1. Platform outcome and component summary
2. Architecture or system map
3. Choose-a-path entry points: hosted, local, self-hosted, SDK, or component development
4. Short start for the most common path
5. Component or package index
6. Compatibility and deployment assumptions
7. Documentation map
8. Support routing by question type
9. Contributing, security, license

Use the README to establish a mental model and route readers. Keep component-level procedures close to each component.

Keep these boundaries:

- explain how components relate instead of presenting a flat feature dump;
- label incompatible getting-started paths separately;
- explain every architecture-diagram label;
- route each question type to the appropriate support channel.

**Accepted when:** a reader can choose the correct operating path, understand how its components relate, complete the common path, and reach component-specific docs or support.
